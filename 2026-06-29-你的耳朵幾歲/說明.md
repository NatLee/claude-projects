# 你的耳朵幾歲？· 高頻聽力測試 🎧

**主題類別**：網路趣聞・冷知識
**產出形式**：A 精緻互動頁（HTML / CSS / JS，純前端、離線可用）
**日期**：2026-06-29

## 這是什麼

一個用瀏覽器就能玩的「高頻聽力小測試」。它用 Web Audio 即時合成純音，從 8 kHz 開始慢慢往上掃，找出你還聽得見的**最高頻率**，再換算成一個好玩的「耳朵年齡」。

人的高頻聽力會隨年齡悄悄退化——這是真實存在的生理現象，醫學上叫**老年性聽損（presbycusis）**。網路上著名的「蚊子鈴聲」就是利用這點：約 17.4 kHz 的聲音，主要只有 25 歲以下的人受得了。這個頁面讓你親耳驗證自己落在哪一段。

## 怎麼玩

直接用瀏覽器打開 `index.html` 即可（離線也能跑，不需網路）。

1. **戴上耳機**，把音量從小慢慢轉大，找個安靜的環境。
2. 按 **「開始自動掃頻」**，聲音會從 8 kHz 一路往上爬。
3. 當你**再也聽不到**聲音的那一刻，立刻按 **「我聽不到了！」**。
4. 頁面會顯示你聽得見的最高頻率，以及對應的耳朵年齡估算與解說。
5. 想針對特定頻率測試，可展開 **「逐音檢查站」**，分別播放 8 k～20 kHz 的純音（含 17.4 kHz 蚊子鈴聲），每格標著「通常幾歲以下聽得到」。超過你這台裝置取樣率上限的格子會自動停用（原因見下方「音訊正確性」）。

你的最佳紀錄會用 `localStorage` 存在本機，下次打開還在。

## 為什麼有趣

把「聽力會隨年齡退化」這種抽象知識，變成你**親耳**就能驗證的體驗。很多人第一次發現自己已經聽不到 17 kHz 時都會愣一下——原來歲月連這種地方都留下了痕跡。拿給不同年齡的家人一起玩，差異特別明顯。

## 背後的原理

- 年輕健康的耳朵聽得到約 **20 Hz – 20 kHz**。
- 耳蝸基底部負責高頻的毛細胞最早磨損，所以聽力是「由高音往低音」逐漸流失，日常人聲（約 0.5–4 kHz）反而保留較久。
- 常見的頻率—年齡對照（近似值，個體差異很大）：
  - 17.4 kHz：約 24–25 歲以下（蚊子鈴聲）
  - 16 kHz：約 30 歲以下（過了 25 歲多半就聽不到 16 kHz 以上）
  - 15 kHz：約 40 歲以下
  - 14 kHz：約 50 歲
  - 12 kHz：約 55–60 歲
  - 10 kHz：約 65 歲
  - 8 kHz：幾乎人人都聽得到

## ⚠️ 重要免責聲明

這只是個**好玩的估算**，**不能取代專業聽力檢查**。結果受耳機／喇叭品質、音量、瀏覽器與環境噪音影響極大——一般筆電與手機喇叭往往根本發不出 16 kHz 以上的聲音，聽不到很正常，**不代表你的聽力有問題**。若你對自己的聽力有任何疑慮，請諮詢耳鼻喉科醫師或聽力師。

## 這個測試「測不準」的地方（誠實交代）

我們在**軟體端**能保證的事：播出去的是乾淨的正弦波、頻率就是畫面上寫的那個數字、沒有爆音、不會超過裝置取樣率上限。

我們**無法**保證的事，也是誤差的主要來源：

- **喇叭／耳機的高頻響應**。多數揚聲器在 16 kHz 以上音壓就大幅衰減，20 kHz 常常根本推不出來。你「聽不到 18 kHz」很可能是喇叭放不出來，不是耳朵的問題。這是本測試無法補償的最大變因。
- **音量沒有校準**。真正的聽力檢查會以校準過的 dB SPL 施測；這裡只能用你自己轉的音量旋鈕。所以本頁測的是「在這個音量下你聽不聽得到」，不是聽力閾值。
- **各頻率音量固定不做等響度補償**。刻意如此：任意加權只會製造另一種偏差，不如維持全頻段等振幅（gain 0.12），並在此說明。
- **反應時間**。自動掃頻時你按下按鈕總會慢個零點幾秒，結果會略微高估（掃頻速率約 545 Hz/秒）。想更精確請改用「逐音檢查站」。
- **環境噪音**會遮蔽高頻。

一句話：把它當成一個**有趣的自我觀察**，不要當成診斷。

## 技術小檔案

- 純前端，無任何外部相依套件；以 Web Audio API 的 `OscillatorNode` 即時合成正弦純音。
- 音量保守（gain ≈ 0.12）並加上淡入淡出，避免高頻爆音。
- 畫面波形以 `<canvas>` 即時繪製：音越高，波形越密。
- 切到背景分頁會自動靜音。

### 音訊正確性（這頁的命脈）

一個聽力測試如果播出去的聲音跟它宣稱的頻率不符，就一文不值。所以有四條硬規則：

1. **只用 `sine`**。方波／鋸齒波含大量泛音，低頻泛音會讓人「以為」自己聽到了高頻。純正弦沒有這個問題。

2. **任何起音／停音／換音都做 gain ramp**（淡入 30 ms、換音淡出 12 ms）。
   直接 `oscillator.stop()` 會在波形上切出一個不連續點，那是一個**寬頻脈衝**——
   它在 100–4000 Hz（任何爛喇叭都放得出來的頻段）注入明顯能量，而乾淨的淡出版本在該頻段幾乎為零。
   這就是聽力測試最經典的假陽性：使用者聽到的是「喀」一聲，卻以為自己聽到了 18 kHz。

   ⚠️ **這裡有個很容易踩的坑：`GainNode.gain` 的預設值是 1，不是 0。**
   為了不讓新舊兩顆音疊在一起，新音會被排在「舊音完全淡出之後」才起音，
   於是 `gain.setValueAtTime(0, t)` 的 `t` 落在**未來**。而依 Web Audio 規格，
   在第一個 automation event 生效之前，`AudioParam` 讀出來的是它的 *intrinsic value*——
   對 `GainNode.gain` 而言就是 **1.0**。
   這代表：若使用者在這短短幾十毫秒內又觸發一次（連點滑鼠，或在按鈕上長按 Enter 讓 OS 自動重複），
   停音程式讀到的音量會是 1.0，`cancelAndHoldAtTime()` 於是把「還沒生效的淡入包絡」整條刪掉、
   改從 1.0 往下 ramp；那顆 oscillator 接著會在這條下降斜線的**中途**才起音——
   **完全沒有淡入的矩形硬切，振幅還高達額定值的 8 倍（+18 dB）**。
   聽起來就是一連串刺耳的「喀」：正好是本條規則要消滅的東西，而且在耳機音量開很大的聽力測試頁上並不安全。
   解法是建立 GainNode 後**立刻 `gain.gain.value = 0`**，把 intrinsic value 先壓成 0；
   如此任何提早發生的 `cancelAndHold` 都只會 hold 在 0，那顆音全程靜音。
   另外還加了兩道保險：音還沒起音就被中止時直接讓它一個取樣點都不發聲；以及音量上限 `min(value, 0.12)`。

3. **可測上限由裝置的取樣率決定**（取 Nyquist 的 95%）。超過 Nyquist 的正弦會 alias 成一個**完全錯誤的低音**。
   常見的 44.1／48 kHz 裝置可以完整測到 20 kHz；但藍牙耳機切到通話模式時取樣率可能掉到 16–32 kHz，
   這時頁面會**誠實下修上限、停用測不了的頻率格**，並在畫面上寫出實際的取樣率與上限——寧可少測，也不騙人。
   若裝置低到連掃頻都做不了（或瀏覽器根本不支援 Web Audio），整個測試會一致地停用，
   不會留下「按了卻毫無反應」的按鈕。

4. **顯示的頻率＝耳朵此刻聽到的頻率**。掃頻整段排在音訊執行緒上（`linearRampToValueAtTime`），
   畫面則從音訊時鐘反推、並扣掉 `baseLatency + outputLatency`（藍牙可達 100–200 ms），
   所以畫面數字不會領先你的耳朵。

各頻率維持等振幅（gain 0.12），不做等響度補償——理由見上方「測不準的地方」。

### 驗證

- 全部 JS 通過 `node --check`。
- 以**符合 Web Audio 規格的 `AudioParam` 模型**（含 intrinsic value、`linearRampToValueAtTime`、
  `cancelScheduledValues`、`cancelAndHoldAtTime` 的語意）＋假 DOM 跑真正的 `app.js`，
  在 48 / 44.1 / 32 / 16 kHz 四種取樣率 × 四種操作情境（正常掃頻、長按 Enter 的鍵盤自動重複、
  滑鼠連點、頻率格連點）下逐一檢查不變式：
  - oscillator 恆為 `sine`；
  - **每顆 oscillator 在起音當下的 gain 必須 === 0**（＝一定有淡入，沒有硬切爆音）；
  - **任何時刻 gain 都不得超過額定的 0.12**（＝不可能出現超大音量）；
  - 每個 `stop()` 之前 gain 都已 ramp 到 0；
  - 同一時間最多只有一顆 oscillator 在響（不疊頻）；
  - 排程頻率從不超過 Nyquist 安全上限（不 alias）；
  - 掃頻中「畫面顯示的 Hz」與「音訊執行緒上該時刻真正播放的 Hz」誤差 < 1 Hz。
