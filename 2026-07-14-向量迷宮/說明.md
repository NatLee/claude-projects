# 向量迷宮 · 向量資料庫其實不做完整搜尋（HNSW）

> RAG 要在幾百萬個向量裡找「最像的那幾個」。大家以為它把每一個都拿來比一遍——其實不是。
> 正式產品（Pinecone、FAISS、Weaviate、Qdrant、pgvector…）幾乎都用 **HNSW** 做**近似**最近鄰搜尋（ANN），
> 用一點點準確率，換來上千倍的速度。這一頁把那座迷宮蓋出來，讓你親眼看貪婪走訪如何一跳一跳穿過圖找到最近鄰。
>
> **隱藏寶藏：向量搜尋是「近似」的，而且這是特性、不是 bug。**

---

## 這是什麼

一個純靜態、可離線的互動頁面。它會在你的瀏覽器裡**當場建一座真的 HNSW 索引**（多層圖），
再**真的跑貪婪圖走訪的查詢**，並把每一步演給你看：從最頂層的入口出發，沿邊跳到更近的鄰居、
逐層下降，最後鎖定最近鄰。旁邊即時算出「這次只比對了幾個點（總共幾個）」——
你會看到近似搜尋掃過的點少得驚人。

沒有預錄動畫、沒有寫死的答案。換個參數，整座圖就重蓋一遍。

## 怎麼玩

1. **放查詢點**：在畫布上點一下，或用鍵盤方向鍵移動準星、按 `Enter` 放下。搜尋動畫會自動播放。
2. **看它怎麼走**：金色路徑就是貪婪走訪的軌跡；被比對過的點會亮起暖色光暈；
   下方字幕會告訴你「現在在第幾層、為什麼下降」。
3. **逐步 / 重播**：用「步進」一步步看，用「重播」再看一次。可切「慢 / 中 / 快」。
4. **暴力對照**：按「🐌 暴力精確搜尋」，畫面會把**全部**的點掃一遍（保證找到真正的最近鄰），
   和 HNSW 比一比：各自掃了幾個點、答案一不一樣。**偶爾 HNSW 會錯過真正的最近鄰**——這正是「近似」的代價。
5. **調參數**：
   - **點數 N**：資料量。調大會看到「掃描比例」往下掉（HNSW 的魔法）。
   - **每點鄰居 M**：每個點在圖上連幾條邊。大一點通常更準、但圖更肥。
   - **建圖 efConstruction**：建索引時每步考慮的候選數。大一點，圖蓋得更好。
   - **查詢 efSearch**：查詢時在第 0 層展開的廣度。這是 recall 與速度的主旋鈕——大一點更準、掃更多。
   - **要找幾個 k**：回傳前 k 個最近鄰。
6. **切圖層**：「疊合 / 第0層 / 上層」——看清「高速公路（上層稀疏長邊）」與「市區道路（第0層稠密短邊）」的分工。
7. **鍵盤**：畫布可用 `Tab` 聚焦。方向鍵移動準星（按住 `Shift` 移大步）、`Enter` 放點搜尋、
   空白鍵播放/暫停、`.`（句號）步進。動態結果會用 `aria-live` 朗讀。

## HNSW 在做什麼

**HNSW = Hierarchical Navigable Small World（階層可導航小世界圖）。** 三個關鍵想法：

### 1. 小世界圖：六度分隔的可導航版
一張圖如果同時有很多**短邊**（連到附近）和少數**長邊**（跳到遠方），任兩點間就會有很短的路徑，
而且只靠「往離目標更近的鄰居走」這種**局部貪婪**就找得到。這是 Kleinberg（2000）把米爾格倫
「六度分隔」變成演算法的洞見，也是 NSW / HNSW 的地基。

### 2. 多層圖：高速公路疊在市區之上
HNSW 把圖疊成好幾層：
- **頂層**只有極少數點、連著很長的邊——像高速公路，幾跳就橫越整個空間。
- **愈往下層點愈多、邊愈短**——像市區巷弄，用來精修。

每個點被放進第幾層是隨機決定的（機率隨層數指數遞減，由 `M` 控制），所以自然形成金字塔。

### 3. 貪婪下降：由粗到細
查詢時，從**頂層入口**開始貪婪走訪：一直往離查詢點更近的鄰居跳，跳到走不動（局部最小）就
**下降一層**，用剛才的落點當新入口，接著走。到最底層（第 0 層）改用寬一點的束搜尋（beam search，
廣度就是 `efSearch`），挑出前 k 個最近鄰。整趟由粗到細，只碰到全圖的一小撮點。

> 建圖時還有一個關鍵：**啟發式選鄰居**。它不是單純挑最近的幾個點連邊，而是偏好「方向多樣」的鄰居——
> 一個候選只有在「離你比離已選的鄰居都更近」時才連。這樣才連得出真正有用的長邊（高速公路），
> 而不是一堆擠在一起的短邊。本頁實作了這個啟發式（論文 Algorithm 4）。

## 為什麼向量資料庫用近似搜尋

### 維度詛咒：精確搜尋在高維會崩潰
真正的嵌入向量是幾百上千維的。在高維空間裡：
- **暴力比對**每查一次都要掃過全部向量，百萬級資料一次查詢就要算幾百萬次距離。
- **樹狀索引（如 KD-tree）**在低維（大約 < 20 維）很快，但維度一高就**退化成暴力掃描**——
  因為要保證找到真正最近鄰、需要檢查的格子數隨維度**指數爆炸**。

於是大家換個思路：**放棄「保證最近」，改求「幾乎最近，但快得多」**。這就是近似最近鄰（ANN）。

### recall vs 速度：一個心甘情願的取捨
衡量近似搜尋準不準，用的是 **recall**：回傳的鄰居裡，有多少真的落在「真正的前 k 名」裡。
`efSearch` 調大 → recall 上升、但掃更多點、變慢；調小 → 更快、但偶爾會錯過。
工程上就在這條曲線上挑一個甜蜜點。

一個代表性的實測（ANN-Benchmarks 類基準，128 維、百萬張圖片向量）：HNSW 可以達到約
**0.94 recall@10**、同時每秒做**數百次查詢**（單執行緒、CPU）——每次查詢只掃過全體的極小比例。
資料愈多，這個比例還會愈小，因為 HNSW 的查詢成本大致隨資料量呈**對數**成長。

## 哪些產品在用 HNSW

HNSW 幾乎成了向量搜尋的預設選項：

- **FAISS**（Meta 的相似度搜尋函式庫，內建 `IndexHNSWFlat`）
- **Weaviate**（以 HNSW 為預設向量索引）
- **Qdrant**（以 Rust 寫成，核心就是 HNSW）
- **Milvus**（支援 HNSW 等多種索引）
- **pgvector**（PostgreSQL 擴充，`0.5.0` 版起加入 HNSW 索引）

你用過的 RAG 系統，底下大概就藏著一座這樣的迷宮。

## 誠實聲明（教學用簡化版）

本頁的 HNSW 是**教學版**：
- 維度只有 **2D**（真實世界是幾百上千維），規模只有**數百個點**（真實是百萬~十億級）。
- 距離用歐氏距離；分層用可重現的亂數，好讓「相同參數建出相同的圖」方便觀察。
- 動畫、配色、字幕都是為了「看得懂」加上去的。

但**多層圖 + 由頂層貪婪下降 + 啟發式選鄰居**這副骨架，和 FAISS、Qdrant 那些真實函式庫是**同一套**。
想看真正的百萬級效能數字，請跑下面「資料來源」裡的基準測試。
