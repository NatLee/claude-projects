為什麼 AI
數不出草莓有幾個 r

你問 ChatGPT「strawberry 有幾個 r」，它常常答錯。不是它笨——是它根本看不到字母。 語言模型讀到的不是一個個 char，而是被切好的 token。這頁讓你當場訓練一個真的 BPE tokenizer， 親眼看文字怎麼被切、又為什麼因此數不出那三個 r。

01

當場訓練一個 BPE

BPE（byte-pair encoding）的規則簡單到不可思議：找出最常相鄰的一對，把它們黏成一個新單位，重複 N 次。選一份語料，按下訓練，看合併一步步發生。

    02

    用你剛訓練的詞表切文字

    輸入任何文字，馬上用剛訓練好的合併規則把它切成 token。每個色塊是一個 token，下面的小數字是它的 id。這就是模型真正吃進去的東西。

    0 個 token 0 個字元 壓縮比 1.0 字元／token

    提示：同一個字在不同詞表下會被切得不一樣——這正是重點。切法是學出來的，不是固定的。

    03

    那個著名的失敗：數 r

    你數 r，用的是字母。模型「數」r，手上只有幾個不透明的 id——字母早就被封進 token 裡看不見了。輸入一個字，兩邊同時數給你看。

    ✎你（用字元數）

    正確答案：0 個 r

    ▦模型（只有 token）

    它看到 0 個 token，沒有一個叫做「r」

    連字母在哪都看不到，是要它怎麼數？

    順帶一提：同一句話，中文比英文貴

    GPT 的 BPE 主要在英文上訓練，所以英文被壓得很短、中文幾乎壓不動。用你目前（多半以英文訓練）的詞表切切看——中文字更少，token 反而更多。

    英文
    I really love eating fresh strawberries.
    0 token · 0 字元
    中文
    我真的很喜歡吃新鮮的草莓。
    0 token · 0 字元

    想反過來？把上面的語料換成中文再訓練一次，中文就會變便宜、英文變貴。誰便宜，取決於詞表在誰身上學的。

    04

    詞表裡的幽靈：glitch token

    既然 token 是統計出來的，就會出意外。2023 年 1 月，Jessica Rumbelow 與 Matthew Watkins 在 SERI-MATS 研究 GPT 的嵌入空間時，挖出一批「幽靈 token」：它們在詞表裡存在，卻幾乎沒被訓練到， 餵給模型就會語無倫次。原因是——這些字串（多為 Reddit 的 r/counting 版使用者名稱、爬蟲殘料）在訓練 tokenizer 的語料裡出現上萬次， 卻在訓練模型的語料裡被濾掉了。有 id、有位置，卻從沒學過該說什麼。

    這幾個是查證過的真實例子（不需重現，OpenAI 後來也修掉了部分行為）。故事最早發表於 LessWrong〈SolidGoldMagikarp (plus, prompt generation)〉，見文末來源。

    05

    從 1994 的壓縮術，到今天的 GPT

    你剛剛跑的演算法，有一條清楚的血緣。

    1. 1994
      BPE 誕生：一個壓縮演算法Philip Gage 在《C Users Journal》發表〈A New Algorithm for Data Compression〉，用「把最常見的相鄰位元組對換成一個新符號」來壓縮資料，效果逼近當時的 LZW。跟語言毫無關係。
    2. 2016
      被帶進 NLPSennrich、Haddow、Birch 在 ACL 論文〈Neural Machine Translation of Rare Words with Subword Units〉裡把 BPE 借來切「子詞」，解決機器翻譯的未登錄詞（OOV）問題——罕見字拆成常見的碎片就不怕沒看過。
    3. 2019
      GPT-2 / GPT-3採用 byte-level BPE，詞表 50,257 個 token（256 個位元組 ＋ 50,000 次合併 ＋ 1 個 <|endoftext|>）。英文平均約 4 個字元擠成 1 個 token。
    4. 2023
      GPT-4換用 cl100k_base，詞表約 10 萬 個 token，程式碼與多語言效率明顯變好。
    5. 2024
      GPT-4o / o1 / o3再換 o200k_base，詞表約 20 萬，對非英文（含中文、emoji）的壓縮再進一步。詞表越大，越多概念能塞進單一 token。

    誠實說

    這頁的 tokenizer 是教學用的小型 BPE：詞表比真實模型小上幾千倍，而且為了看得清楚，我用「字元」當基本單位；真實 GPT 用的是「256 個位元組」當基本單位，所以罕見中文字和 emoji 常常會爆成好幾個 token。 但合併的演算法、以及切分時「照學到的順序優先合併」的邏輯，跟 GPT 用的是同一套。你剛剛親手跑的，就是那個核心。
