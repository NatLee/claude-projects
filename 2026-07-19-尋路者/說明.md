# 尋路者 · A* 尋路演算法

一個**真的在瀏覽器裡跑演算法**的網格尋路沙盒。拖牆造出你自己的迷宮，讓 BFS、Dijkstra、貪婪最佳優先（Greedy Best-First）、A* 四種演算法在同一張地圖上找路——不是預錄動畫，逐格探索的過程、最後的路徑，全是當場即時運算出來的。

## 這是什麼

尋路（pathfinding）是「在地圖上找出從 A 到 B 最短路徑」的問題，從電玩 NPC 走路、機器人與自駕車規劃路線，到地圖軟體算路線，底層幾乎都是本頁介紹的這幾種演算法（或它們的變形）。本頁讓你：

- **親手畫一張地圖**：拖曳造牆、擦牆，拖曳起點／終點到任何位置。
- **看演算法怎麼「思考」**：逐格動畫呈現「邊界（open，正在考慮）」與「已探索（closed，已經確定最短距離）」的擴散過程，最後把找到的路徑亮出來。
- **四種打對台**：BFS、Dijkstra、貪婪最佳優先、A* 可以單獨跑，也可以四個並排一次跑完、看數據表。
- **調一個滑桿，看 A* 變回 Dijkstra**：A* 的啟發式權重滑桿拉到 0，A* 會在你眼前退化成 Dijkstra——因為 Dijkstra 本來就是 A* 在「沒有方向感（h=0）」時的特例。

## 怎麼玩

1. **畫地圖**：在棋盤空白格上按住拖曳＝畫牆；在已有的牆上拖曳＝擦牆。直接拖曳綠色起點／橘色終點可以移動它們（牆會自動讓開）。
2. **選演算法**：上方切換 BFS／Dijkstra／貪婪最佳優先／A*，按「▶ 開始」看逐格動畫；「動畫速度」滑桿可以調快調慢，拉到底幾乎是瞬間看到結果。
3. **調權重滑桿看 A* 退化**：只有選到 A* 時這個滑桿才有作用（並列比較模式下永遠作用在 A* 那一格）。w = 1 是標準 A*；拉到 **w = 0**，A* 就會和 Dijkstra 探索出一模一樣的格子（因為公式完全相同）；拉到 w > 1 是「加權 A*」，更快但不再保證最短。
4. **🎲 隨機迷宮**：在目前的起訖點周圍重新灑一批牆（保證仍然連通）。
5. **🪤 貪婪陷阱**：載入一張刻意設計、會讓貪婪最佳優先「上當」的地圖——它會很有自信地衝進一個看起來越來越近、實際上要繞遠路的口袋。
6. **🧹 清空**：清掉所有牆。
7. **並列比較四種**：切換到這個模式、按「🎬 開始比較」，四張棋盤同時開跑，下方數據表即時填入探索格數、路徑長度、是否最短。
8. **鍵盤操作**：棋盤本身可以用 <kbd>Tab</kbd> 選取後，用方向鍵移動游標、<kbd>空白鍵</kbd> 切換牆壁、<kbd>S</kbd>／<kbd>E</kbd> 把起點／終點移到游標位置。

## 為什麼有趣（這裡是哇時刻）

用本頁**內建的預設地圖**實際跑一次：

| | 探索格數 | 路徑長度 |
|---|---|---|
| Dijkstra | 204 格 | 25 步 |
| A*（w=1） | **36 格** | 25 步 |

同一張地圖、同樣找到 **25 步**的最短路徑，Dijkstra 得把起點附近像水波一樣往四面八方摸過一輪（一個不管方向的「圓」），A* 卻幾乎是筆直朝終點探——只探索了 Dijkstra 的**不到兩成**。把權重滑桿拉到 0，A* 立刻變回那個「不管方向的圓」，探索格數當場跳回 204。這正是「方向感」這件事的份量。

再用**內建的貪婪陷阱地圖**看貪婪最佳優先誠實地失手：

| | 路徑長度 |
|---|---|
| 最短路徑（BFS／Dijkstra／A* 一致） | 27 步 |
| 貪婪最佳優先 | **51 步**（多繞了 24 步，將近兩倍）|

貪婪最佳優先只看「感覺離終點多近」，完全不管已經走了幾步——一頭栽進一個看起來很有希望、實際是死路的口袋，等它探到底退出來，走的已經不是最短的那條路了。A* 在同一張地圖上仍穩穩找到 27 步，因為它用 g(n) 老實記帳，不會被「感覺快到了」騙走。

## 方法論／實作細節

**四方向移動＋曼哈頓距離啟發式**：本頁只允許上下左右移動（不含對角線），啟發式函式 h(n) 用曼哈頓距離（水平格數＋垂直格數之和）估計到終點的距離。

- **可採納（admissible）**：牆只會讓實際路徑變長、不會變短，所以曼哈頓距離必定 ≤ 真正還要走的步數，天生滿足 admissible——這保證 A* 找到的路徑就是最短路徑。
- **一致（consistent／monotone）**：網格上每走一步，曼哈頓距離最多剛好改變 1，和那一步的成本（1）完全對齊，滿足 h(n) ≤ cost(n,n′) + h(n′)。這保證格子一旦被關閉（closed），最短距離就已經定案，A* 不必回頭重開任何格子。
- 若改成八方向移動，曼哈頓距離就不再 admissible（對角線一步的曼哈頓估計是 2，但實際成本可能設為 1 或 √2），必須換成對角距離（Chebyshev／octile distance）才能維持 admissible／consistent；本頁刻意只做四方向，讓證明與實作單純、乾淨。

**真的實作二元堆積（binary heap）優先佇列**：`MinHeap` 是標準的陣列式二元最小堆，`push`／`pop` 都是 O(log n)，用來實作 Dijkstra、A*、貪婪最佳優先的優先佇列（BFS 則用一般 FIFO 佇列，符合它的定義）。已用測試腳本驗證：對堆積推入大量亂數後連續 `pop()`，取出的序列必為非遞減——這是二元堆積正確性的核心保證。

**Dijkstra 與 A* 共用同一段程式碼**：兩者的排序依據都是 f(n) = g(n) + w·h(n)，差別只在權重 w。Dijkstra 就是把 w 固定成 0 呼叫同一個函式；A* 是把 w 設成使用者調的滑桿值（預設 1）。這不是「模擬相似效果」，是**同一份程式碼、同一個參數**——所以 w 拉到 0 時，A* 的探索序列與 Dijkstra 逐格完全相同，不是巧合而是數學上的必然。

**貪婪最佳優先的誠實實作**：只用 h(n) 排序，且一個格子一旦被「發現」（第一次進入邊界），之後就算找到更短的路徑也不會重新指定它的父節點（不 relax）。這是貪婪最佳優先在教科書上的標準定義，也正是它有時找不到最短路徑的根本原因——本頁沒有刻意調弱它，是它本來就長這樣。

**逐格動畫怎麼來的**：每次搜尋會先完整、同步地跑完整個演算法（對這個大小的棋盤是毫秒級的），得到「關閉順序」；動畫播放時，用這個順序重建每一步「有哪些鄰居格第一次被摸到（frontier）」「哪一格被關閉（closed）」，再用單一 `requestAnimationFrame` 迴圈依「動畫速度」逐格套上對應的 CSS class 播放出來。分頁切到背景時（`document.hidden`）迴圈直接停止排程，回到前景由 `visibilitychange` 事件重新啟動；`prefers-reduced-motion` 開啟時改為瞬間套用最終結果，並透過 `matchMedia` 的 `change` 事件動態監聽（不必重新整理頁面）。

**迷宮怎麼生成**：用 mulberry32 這顆小型可種子（seedable）的偽隨機數產生器，在起訖點周圍保留淨空區後，以指定機率灑牆，並用 BFS 檢查起訖點是否仍連通；不連通就用同一顆種子繼續往下試，最多 60 次。首頁預設地圖與「貪婪陷阱」地圖都是用**固定種子**產生，所以每次打開頁面看到的初始地圖、以及上面表格列出的數字都會一致；按「🎲 隨機迷宮」則改用 `Math.random()` 每次產生不同地圖。

## 演算法對照表

| 演算法 | 使用資訊 | 保證最短？ | 時間複雜度 | 定位 |
|---|---|---|---|---|
| BFS | 只看步數（層數） | ✓ 邊權相同時 | O(V+E) | 無權重網格最基本的地毯式搜尋 |
| Dijkstra | g(n) 實際成本 | ✓ 任何非負權重 | O((V+E) log V) | 不看方向，穩紮穩打由近到遠 |
| 貪婪最佳優先 | h(n) 預估距離 | ✗ 不保證 | O(E log V)（實務常更快） | 只認方向，快但可能被騙 |
| A* | g(n) + h(n) | ✓ h 為 admissible 時 | 最壞同 Dijkstra，實務展開節點數少很多 | 又快又準——方向感加上紀律 |

（V 為格子數，E 為邊數，四方向網格中 E 約為 4V）

## 技術備註

- 單一自包含 `index.html`（HTML + CSS + JS 全在同一檔），可直接雙擊開啟，零外部資源（不 fetch、不用外部字型、不接 CDN）。
- 尋路引擎（二元堆積、BFS、Dijkstra／A* 共用函式、貪婪最佳優先、迷宮生成）為純函式、不碰 DOM，已抽出成獨立 Node.js 腳本，在**超過 1,200 張隨機地圖**上跑過斷言測試：BFS／Dijkstra／A* 的最短路徑長度永遠一致、A* 探索格數永遠不超過 Dijkstra、w=0 時 A* 與 Dijkstra 逐格探索序列完全相同、貪婪最佳優先確實會在其中約四分之一的地圖上找到比最短路徑更長的路。二元堆積另外用大量亂數 push／pop 驗證取出序列必為非遞減。
- `<script>` 內容已抽出並以 `node --check` 驗證語法正確。
- 動畫只用 `transform`／`opacity`，`requestAnimationFrame` 迴圈在分頁隱藏時停止排程；`prefers-reduced-motion` 以 `matchMedia` 動態監聽並完整降級為瞬間顯示結果。
- 鍵盤可完整操作（棋盤本身支援方向鍵、空白鍵、S／E 鍵），互動元素具 aria-label，搜尋結果透過 aria-live 播報。
