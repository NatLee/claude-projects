# 每日子頁面 Cowork Prompt

> 給每日排程（Claude cowork schedule）使用的 prompt。整個工作資料夾是一個「大專案」：
> 一個純靜態網站，入口是根目錄的 `index.html`（索引室：時間軸卡片牆首頁），每天為它長出一個新的子頁面。

---

目標：每天早上自動為使用者的「每日小專案」靜態網站（位於 C:\Users\NatLee\Desktop\claude-projects，已是 git 倉庫，入口為根目錄 `index.html`）新增一個全新、好玩、有質感的**子頁面**，帶來驚喜；並像一位長期協作的同事一樣，順手保養整個網站。全程使用繁體中文。

【大專案架構】整個資料夾是一個可直接部署（如 GitHub Pages）的純靜態網站：

- 根目錄 `index.html`：首頁（索引室：今日之星 hero ＋ 時間軸卡片牆，含即時搜尋、類別篩選、時間／分類分組）。專案清單寫在標示清楚的 `const PROJECTS = [...]` 陣列裡，卡片、統計數字與類別籌碼皆由陣列自動生成。
- 每個子頁面一個資料夾，**依月份分組**：`projects/YYYY-MM/YYYY-MM-DD-簡短專案名/`，入口一律是該資料夾內的 `index.html`，並附一份 `說明.md`。
- 根目錄只保留 `index.html`、`README.md`、`PROMPT.md` 與 `projects/`，不要在根目錄直接建立專案資料夾。
- 全站禁止任何後端與資料庫：需要儲存資料時一律使用 localStorage。

【主題類別】每天從以下六大類「隨機」挑選一類（盡量輪流、避免連續重複；參考 README 目錄近期紀錄）：

1. 網路趣聞・冷知識
2. 奇聞軼事（歷史謎團、都市傳說查證、不可思議的真實故事）
3. 科學趣聞（太空、生物、物理的驚奇現象與實驗）
4. 學習新知
5. 生活痛點小工具
6. 創意・娛樂

【產出形式】一律是純靜態子頁面（HTML/CSS/JS），但**風格與互動形式每天都要有變化**：

- 可以是多檔頁面（index.html + style.css + app.js）或單檔自包含頁面。
- 可從 cdnjs 載入函式庫；除此之外不得依賴任何外部資源（頁面必須自包含、可直接開啟）。
- 需要儲存使用者資料時使用 localStorage，所有 key 必須加該專案專屬的英文短前綴（例如 `gacha.`、`fermi.`），避免同網域下互相覆蓋。
- 互動形式要輪替：遊戲、模擬器、視覺化、工具、收集系統、文字冒險、聲音玩具……不要連續幾天同一型。

【產出數量】預設每天做 **1 個**子頁面。若排程或指令指定要做多個（例如「今天做 3 個」「多做幾個」），則把步驟 2～7 重複 N 次，每次做出一個完整、獨立的子頁面，並遵守以下規則：

- **命名不變**：資料夾一律 `projects/YYYY-MM/YYYY-MM-DD-簡短專案名`（月份資料夾若不存在就建立）。同一天的多個專案靠「專案名」區分即可（例如 `2026-07-13-真實大小`、`2026-07-13-聲音地圖`），**不要加流水號**（不要 `-01`、`-02`）——顯示順序由首頁 `PROJECTS` 陣列決定，不是資料夾排序。倉庫已有先例：`2026-06-26` 底下就有兩個專案。
- **同一批之間也要不一樣**：這 N 個必須分屬**不同類別**、採用**不同互動形式**、題材彼此無關。步驟 1 的「避免重複」不只要對照歷史，也要對照本批已完成的專案。
- **localStorage 前綴互異**：同批各專案的 key 前綴不得相同或相近。
- **誰當今日之星**：首頁 hero 與「今日之星」徽章只給 `PROJECTS` 陣列的**第一筆**。所以整批做完後，把你**最滿意的那一個**排到陣列最上方（其餘同日作品依序排在它下面），不要單純依完成順序決定。
- **保養只做一次**：步驟 8 的「順手保養」整批只做一次，不要每個專案都做。
- **git 節奏**：每個專案各自一個 commit（訊息格式同步驟 9）。**不要 push**（使用者會自己推）。
- **品質不打折**：寧可少做一個，也不要為了湊數量交出達不到下述設計品質標準的作品。

【設計品質標準】全站已整體重整過，新子頁面必須達到同一水準——「獲獎作品集網站」的工藝，fancy 但不吵雜：

- 清楚的視覺層級與節奏（留白、對齊、字級階梯），桌面善用空間、手機 <560px 單欄流暢。
- 進場編排：開頁元素依序優雅入場（stagger），總長 ≤1.2 秒。
- 微互動：按鈕/卡片的 hover、按下、聚焦狀態；數字滾動；狀態切換過渡。
- 至少一個貼合主題、令人記住的「哇」時刻（揭示、慶祝、翻牌、光效……）。
- 動畫紀律：只用 transform/opacity、rAF 節流、canvas 與 rAF 迴圈在分頁隱藏或離屏時暫停、`prefers-reduced-motion` 完整降級（matchMedia change 動態監聽）。
- 無障礙：鍵盤可操作、聚焦樣式、互動元素有 aria-label、動態結果用 aria-live。
- 頁面放一個低調、貼合主題的「← 回首頁」連結指向 `../../../index.html`（子頁在 `projects/YYYY-MM/專案名/` 底下，距根目錄三層）。
- **註腳要克制**：頁面 footer 最多一到兩行，是作品的收尾，不是免責聲明區。**不要寫 localStorage／隱私／離線的注意事項**（「資料只存在你的瀏覽器」「不會上傳到任何地方」「全程離線可用」之類）——一般使用者不需要知道，寫了只顯得囉唆；也不要在頁面上叫使用者去看 `說明.md`。涉及事實或數據時，只留**一行**極短的來源掛名（列出處名即可，如「資料來源：NASA、Wikipedia」，不要列文章標題），完整來源與方法論一律寫進 `說明.md`。唯一例外是**真正攸關使用者信任的告知**（例如請人輸入密碼的頁面要說明密碼不會離開瀏覽器、健康／聽力測驗要說明不能取代專業診斷），這種留著，但也要短。

【執行步驟】

1. 避免重複：列出 `projects/` 底下既有的月份與專案資料夾，並瀏覽 `index.html` 裡的 `PROJECTS` 陣列（那是唯一的作品清單來源），看看最近做過的類別與互動形式，刻意挑最近沒做過的組合、全新的題材。（若本次要做多個，每一個也都要避開**本批稍早已完成**的類別、形式與題材。）

2. 構思點子：挑一類主題，想一個具體、有趣、做得完的點子。凡屬「網路趣聞」「奇聞軼事」「科學趣聞」「學習新知」，先用 WebSearch 找一兩個真實素材或數據當靈感，確保內容正確、有料。

3. 建立子頁面：建立 `projects/YYYY-MM/YYYY-MM-DD-簡短專案名`（日期用今天，台灣時區；月份資料夾不存在就一併建立），放入 `index.html` 與所需檔案，達到上述設計品質標準。

4. 測試（很重要）：
   - 抽出所有 JS（含內嵌 `<script>`）跑 `node --check`。
   - 確認直接雙擊開啟可完整運作：相對路徑正確、無 fetch 後端呼叫、外部資源只來自 cdnjs。
   - localStorage key 都有專案前綴。
   修正所有明顯問題，交付「能跑」的成品。

5. 附上說明：資料夾內放 `說明.md`——這是什麼、怎麼玩／用、為什麼有趣或實用、完整資料來源（若涉及事實）；若有用 localStorage 在此註明。冗長的來源清單、方法論、隱私與儲存說明**都住在這裡**，不要搬到頁面上（見設計品質標準的「註腳要克制」）。

6. 掛上首頁：編輯根目錄 `index.html` 的 `PROJECTS` 陣列，把今天的專案**加在陣列最上方**（欄位照既有格式：`date`、`title`、`emoji`、`dir`、`category`、`desc`；**`dir` 要寫完整相對路徑 `projects/YYYY-MM/YYYY-MM-DD-專案名`**；`category` 必須與六大類字串完全一致）。不要動陣列以外的程式碼——卡片牆會自動長出新卡片、今日之星 hero 會自動換成陣列第一筆的作品。（多產出時：整批做完後，把**最滿意的那一個**移到陣列最上方，讓它成為今日之星。）

7. 不需更新 `README.md`：根 README 已精簡為說明文件，**不再維護作品清單**——清單唯一來源是 `index.html` 的 `PROJECTS` 陣列（步驟 6 已完成）。只有在流程或結構本身改變時才需要動 README。

8. 順手保養（cowork 精神，時間盒 15 分鐘內）：挑一個「最舊的、最近沒被保養過」的既有子頁面快速健檢——抽 JS 跑 `node --check`、點檢主要互動的接線、看有無明顯 bug。發現小問題就修（不改玩法、不改 localStorage key）；沒有就跳過。在 commit 訊息註明保養了哪個專案。

9. 整理 git（**只 commit、不 push**——使用者會自己推）：先 `git status` 檢視所有變更，把它們整理成一個乾淨、有條理的 commit（今天的子頁面、首頁 PROJECTS、以及保養修正都納入；若動到不相干的暫存檔或殘留，先排除或清掉再提交）。commit message 用繁體中文，格式如「新增子頁面：YYYY-MM-DD 專案名（類別）」，並在內文條列今天做了什麼（新增專案、掛上首頁、保養了哪個舊專案）。提交後確認 `git status` 是乾淨的，把工作樹留在可直接 push 的狀態。
   - 多產出時：**每個專案各自一個 commit**（各自帶上它的首頁與 README 變更），保養修正併入最後一個 commit 或獨立一個。
   - 若提交前發現有 `.git/*.lock` 殘留（多為前一次 git 背景維護被中斷留下），確認沒有 git 程序在跑後移除再提交。

10. 呈現：用 present_files 秀出今天子頁面的 `index.html`，一兩句話介紹今天的驚喜；若有保養既有專案，一句話帶過。

【風格與原則】

- 全程繁體中文。
- 每天都要「不一樣」：主題、風格、互動方式都要變化，但都活在同一個靜態網站裡。
- 小而完整：一天內做得完並測得過，能用、沒有明顯 bug。
- 真實正確：凡涉及事實或數據務必查證（WebSearch）。
- 全站一致的底線：純靜態、自包含、localStorage 加前綴、掛上首頁與 README、回首頁連結、達到設計品質標準——其餘盡情發揮。
- 重點是趣味、美感與實用，讓使用者每天早上打開首頁時眼睛一亮。
