# 知識庫下毒・PoisonedRAG 實驗室

一座可以親手「下毒」的 RAG 問答系統。你會先看它乾淨地答對，接著往知識庫塞幾筆動過手腳的文件，
看答案當場翻成攻擊者要的錯誤答案，再打開防禦把同一份毒文件擋下。這是資安教育，用**無害情境**示範漏洞的形狀。

## 這是什麼

RAG（Retrieval-Augmented Generation，檢索增強生成）讓大型語言模型在回答前，先去讀一個「外部知識庫」，
再根據檢索到的內容作答。好處是知識可更新、幻覺變少。但這也開了一扇後門——
**攻擊者不需要碰模型，只要往知識庫塞幾筆動過手腳的文件，就能讓 RAG 自信地講出攻擊者指定的答案。**
這種攻擊叫 **PoisonedRAG（知識庫下毒 / 知識腐蝕攻擊）**。

本頁的 RAG 靶子是用純 JavaScript 寫的：
- **檢索是真的**：正確實作的 BM25 與 TF-IDF 餘弦相似度，你可以切換比較。
- **生成是抽取式示範**：真實 RAG 這一步會把檢索到的 top-k 文件餵給 LLM 生成；這裡改用抽取式——
  直接挑「最相關文件裡最相關的那一句」當答案。刻意如此，就是要讓你看見：**答案完全取決於檢索到什麼。**

情境是虛構品牌「霧島珈琲」智慧手沖機 M1 的產品 FAQ，下毒效果只是把「退貨 7 天」變成「999 天」這類無害但錯誤的答案。

## 怎麼玩

1. **看乾淨狀態**：在中間點一個預設問題（例如「退貨期限是幾天？」），右邊會給出正確答案，
   並在左邊知識庫把被檢索到的文件亮起、標上排名與分數。
2. **下毒**：到「下毒工作台」——
   - 一鍵注入預設毒文件，或
   - 自己撰寫：填「① 檢索誘餌」（問題關鍵字）＋「② 目標錯誤答案」，看合成預覽，按「注入知識庫」。
3. **重新提問**：對同一題再問一次，看答案當場翻成攻擊者要的錯誤答案，毒文件在檢索結果裡「贏過」真文件。
4. **看佔比**：面板顯示「知識庫有 N 筆，你只加了 X 筆毒文件」，佔比極小，攻擊卻成功了。
5. **開防禦**：切換三個防禦開關，看同一份毒文件被擋下或被稀釋；也試試「隱蔽模式」看它如何繞過困惑度過濾。
6. **自由沙盒**：可加入自訂可信文件、清除毒文件、全部重置，自己玩。

## PoisonedRAG 的兩個條件（缺一不可）

一份成功的毒文件要同時滿足：

1. **檢索條件（retrieval condition）**：毒文件必須先被檢索到，才有機會影響答案。
   所以毒文件要**嵌入攻擊者問題的關鍵字**，讓它在「那一題」上排進 top-k。
2. **生成條件 / 有效性條件（effectiveness condition）**：被檢索到之後，內容要能**誘導出目標答案**。

PoisonedRAG 的做法是把一份毒文件**拆成兩截**，一截負責被檢索到、一截負責誘導答案。
本頁的「撰寫毒文件」介面刻意把這兩截標成不同顏色，讓你看見這個結構。

## 為什麼「極少數文件」就夠？

- **Top-k 是一扇窄門**：模型只讀檢索排進前 k 名的少數文件。你不必贏過整個資料庫，
  只要在目標問題上擠進前幾名。一筆精準命中關鍵字的毒文件，往往就能擠掉真文件。
- **模型太相信檢索內容**：RAG 的設計就是「以檢索到的內容為準」，毒文件贏了檢索，答案就跟著翻。
- **論文數據**：Zou 等人（2024）report——只要在一個有**數百萬筆**的知識庫裡，
  為每個目標問題注入**約 5 筆**毒文件，PoisonedRAG 就能達到約 **90%** 的攻擊成功率
  （NQ 97%、HotpotQA 99%、MS-MARCO 91%，以 PaLM 2 為例）。極少數，就夠。

本頁的量化面板是縮小示範：乾淨知識庫 20 筆，注入 1 筆毒文件（約 4.76%）就足以翻盤。

## 防禦：能擋，但別高枕無憂

本頁的三個開關對應三種常見防禦：

1. **來源白名單**：只檢索可信來源的文件，外部注入直接出局。
   *繞過*：一旦攻擊者攻陷「可信來源」本身（供應鏈、開放編輯的維基、被駭的網站），白名單就失效。
2. **困惑度（perplexity）過濾**：用語言模型量文本「讀起來多順」，過濾異常拼湊的文字。
   關鍵字硬塞的毒文件會被抓；但論文指出，用 LLM 寫得很通順的毒文件困惑度很低——
   試試本頁的「隱蔽模式」（把毒文件擬態成正常文件、只換掉數字），它就溜過去了。
3. **知識庫擴充 + 多數決**：多取幾筆、讓多數的真文件把單筆毒文件稀釋掉。
   *繞過*：這只是抬高門檻。攻擊者多塞幾筆湊成多數（論文注入 5 筆去主導 top-k），一樣翻盤。
   本頁實測：1 筆毒文件擋得住，注入約 3 筆即可打敗多數決。

**誠實說**：這些防禦都能被更強的攻擊部分繞過。真正的重點是——
**檢索資料的來源必須可信，且系統不能無條件相信檢索到的內容。**

## 與 indirect prompt injection 的關係

PoisonedRAG 其實是 **indirect prompt injection（間接提示注入）** 的一種：
把指令或錯誤資訊藏進「模型會讀到的資料」裡（網頁、PDF、郵件、被 RAG 檢索到的文件），
而不是直接打進使用者的輸入框。它比一般間接注入多了一道門檻——**內容必須先被檢索到**——
這正是「檢索條件」存在的原因。這也連結到 RAG 的其他失敗模式：檢索到過時或矛盾的資料、模型過度相信檢索內容。
在 OWASP 的 GenAI 風險清單中，這歸類於 **LLM01: Prompt Injection**。

## 誠實聲明

- **規則式模擬，非真實 LLM。** 全程不呼叫任何 AI／LLM API。檢索是真的（BM25、TF-IDF 餘弦），
  生成是抽取式示範（直接挑句子）。
- **無害情境。** 知識庫是虛構咖啡機 FAQ，下毒效果只是無害的錯答案。目的是理解漏洞機制、學會怎麼防，
  **不是**提供攻擊真實系統的工具。
- **規模是示意。** 瀏覽器裡放不下數百萬筆；佔比面板是縮小版，論文的真實規模寫在對照文字裡。

## 技術說明

- 純靜態、**零外部資源**、雙擊 `index.html` 即可離線運作。無 `fetch`、無追蹤、不呼叫任何 AI API。
- 動畫只用 transform／opacity；背景 canvas 迴圈在分頁隱藏或離屏時暫停；
  完整支援 `prefers-reduced-motion`（並提供「減少動態」開關）。
- 檔案：`index.html`、`style.css`、`app.js`、`說明.md`。
- 測試：`poison-test.js`（放在專案資料夾外的 outputs），驗證乾淨答對、下毒翻盤、防禦擋下與佔比量化。

## localStorage 說明

本頁只用瀏覽器本機儲存，所有 key 一律加前綴 `poison.`：

- `poison.lab.v1`：你的自訂內容與設定（新增的可信文件、毒文件、檢索器、k 值、防禦開關、上次的問題）。
- `poison.reducedMotion`：是否手動開啟「減少動態」。

這些資料只存在你的瀏覽器、不會上傳。按「全部重置」或清除瀏覽器資料即可移除。
