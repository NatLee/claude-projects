# 草莓的 R 🍓 — 為什麼 AI 數不出 strawberry 有幾個 r

一個可以**當場訓練一個真的 BPE（byte-pair encoding）tokenizer** 的互動網頁。
你會親眼看見文字如何被切成 token，並理解那個著名的失敗：語言模型為什麼數不出草莓有幾個 r。

---

## 這是什麼

大型語言模型（GPT、Claude…）**看不到字母**。輸入文字前，會先被一個叫做 **tokenizer** 的東西
切成一塊塊的 **token**，再換成一串數字（token id）。模型從頭到尾看到的都是這些 id，
從來沒有「一個字母 r」這種東西。

這頁把整個過程攤開來，全部在你的瀏覽器裡即時運算，沒有任何後端、沒有連線：

1. **訓練區** — 選一份語料（或貼你自己的文字），按下「訓練」，純 JavaScript 當場跑 BPE：
   統計最高頻的相鄰字元對 → 合併成一個新單位 → 重複 N 次。**每一步合併都看得見**
   （例如「第 12 次：`e` ＋ `r` → `er`，出現 143 次」）。合併次數＝詞表大小，可自由調整。
2. **切分區** — 輸入任何文字，即時用剛訓練好的詞表切成 token，用彩色色塊呈現，附上每個 token 的 id，
   並算出「X 個 token、Y 個字元、壓縮比多少」。
3. **數 r 小遊戲** — 輸入 `strawberry`，左邊用**字元**數出正確答案（3 個 r），
   右邊顯示模型看到的其實是幾個**不透明的 token**。你會很直觀地懂：它連 r 在哪都看不到，怎麼數？
4. **語言成本對照** — 同一句話，中文 vs 英文的 token 數。用（多半以英文訓練的）詞表切切看，
   你會發現中文字更少，token 反而更多。
5. **glitch token 資訊卡** — 詞表裡的「幽靈 token」故事，附查證過的真實例子。

---

## 怎麼玩

- **先按「訓練 tokenizer」**（頁面載入時也會自動先訓練一次）。拉動「合併次數」滑桿再訓練一次，
  看 token 怎麼隨著詞表變大而變長、變少。
- 到**切分區**打字，看它即時被切開。試試把同一個字用不同詞表切，結果會不一樣——這正是重點。
- 到**數 r 小遊戲**輸入 `strawberry`、`raspberry`、你的名字、任何字，比較「你數的」與「模型能看到的」。
- 想推翻「中文比較貴」？把訓練語料換成「中文」再訓練一次，中文就會變便宜、英文變貴。
  誰便宜，取決於詞表在誰身上學的——**這正是 GPT 對中文比較不划算的真正原因**（它主要在英文上訓練）。

## 為什麼有趣（它跟 GPT 的關係）

- **strawberry 為什麼數不出 r**：模型不是看到 `s-t-r-a-w-b-e-r-r-y` 十個字母，而是看到少數幾個
  token（依詞表而定，例如 `st｜raw｜berry`，甚至整個併成一個 token）。每個 token 只是一個 id，
  「裡面有幾個 r」這件事，在轉成數字的那一刻就消失了。所以計數任務對它天生困難。
- **9.11 為什麼看起來比 9.9 大**：多位數字的切法很不直覺（`9.11` 和 `9.9` 會被切成不同的 token），
  這是原因**之一**。但老實說，這不是全部——就算改用「一位數字一個 token」的模型也還是會錯，
  所以還牽涉到模型本來就不是為算術設計的、以及版本號／日期的先驗（軟體世界裡 9.11 確實比 9.9 新）。
- **同樣的演算法**：這頁用的 BPE，和 GPT 用的是**同一套**——統計最高頻 pair、合併、重複；切分時
  照「學到的先後順序」優先合併。差別只在規模與基本單位（見下方「誠實說」）。

### 從壓縮術到 GPT（查證過的脈絡與數字）

| 年份 | 事件 |
|---|---|
| **1994** | Philip Gage 在《C Users Journal》發表〈A New Algorithm for Data Compression〉，BPE 原本是**資料壓縮**演算法，跟語言無關。 |
| **2016** | Sennrich、Haddow、Birch 在 ACL 論文〈Neural Machine Translation of Rare Words with Subword Units〉把 BPE 帶進 NLP，用子詞解決未登錄詞（OOV）問題。 |
| **2019** | GPT-2／GPT-3 採 byte-level BPE，詞表 **50,257** 個 token（256 位元組 ＋ 50,000 合併 ＋ 1 個 `<|endoftext|>`）。 |
| **2023** | GPT-4 改用 `cl100k_base`，詞表約 **10 萬**。 |
| **2024** | GPT-4o／o1／o3 改用 `o200k_base`，詞表約 **20 萬**，對中文與 emoji 的壓縮更好。 |

### glitch token（幽靈 token）

2023 年 1 月，Jessica Rumbelow 與 Matthew Watkins 在 SERI-MATS 研究 GPT 嵌入空間時，
發現一批像 ` SolidGoldMagikarp`、` petertodd`、` TheNitromeFan` 的 token：它們**在詞表裡存在**，
卻幾乎**沒被模型訓練到**，餵給模型會產生詭異行為。原因是這些字串（多為 Reddit「r/counting」數數版的
使用者名稱、爬蟲殘料）在**訓練 tokenizer 的語料**裡出現上萬次，卻在**訓練模型的語料**裡被濾掉了。
故事發表於 LessWrong〈SolidGoldMagikarp (plus, prompt generation)〉。OpenAI 後來修掉了部分行為。

---

## 誠實說

這頁的 tokenizer 是**教學用的小型 BPE**：

- 詞表比真實模型小上**幾千倍**（這裡通常幾十～幾百個 token，GPT 是 5 萬～20 萬）。
- 為了看得清楚，這裡用「**字元**」當基本單位；真實 GPT 用「**256 個位元組**」當基本單位，
  所以罕見中文字和 emoji 在真實模型裡常常會爆成好幾個 token（這頁把單一 emoji 當成一個 token，是簡化）。
- 但**合併的演算法、以及切分時「照學到的順序優先合併」的邏輯，跟 GPT 用的是同一套**。
  你親手跑的，就是那個核心。

## 技術與隱私

- **純靜態、零外部資源**：不引用任何 CDN／字型／圖片，一切自己畫。**雙擊 `index.html` 即可離線運作**，
  沒有任何 `fetch`。
- **localStorage**：只用來記住你的偏好，key 一律加前綴 `tok.`：
  - `tok.corpus` — 選的內建語料
  - `tok.merges` — 合併次數（滑桿值）
  - `tok.customCorpus` — 你貼的自訂語料
  - `tok.segText` — 切分區的輸入
  - `tok.rWord` — 數 r 遊戲的輸入
  這些只存在你自己的瀏覽器裡，不會上傳到任何地方。清除瀏覽器資料即可全部移除。
- **無障礙**：可完整鍵盤操作、有聚焦樣式與 `aria-label`，動態結果用 `aria-live` 播報；
  完整支援 `prefers-reduced-motion`（會即時關掉動畫）。
