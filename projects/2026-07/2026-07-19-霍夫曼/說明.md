# 霍夫曼編碼 · 常用字給短碼

## 這是什麼

一個**真的能用**的文字壓縮工具。貼上一段文字，它會：

1. 統計每個字元出現的頻率。
2. 用真正的優先佇列（最小堆積）**由下而上**長出一棵霍夫曼樹——每一步都取出目前權重最小的兩個節點合併成新節點，動畫會逐步播放這個合併過程。
3. 產生每個字元對應的 0/1 前綴碼，常用字自動拿到最短的碼。
4. 把文字真的編碼成一串 0/1 位元，再解碼回文字，驗證「還原成功」。
5. 把「原始固定長度編碼」「霍夫曼壓縮後」「夏農熵理論下界」三個數字放在一起比較，算出壓縮率。

不是查表、不是預先算好貼上去的數字——頁面上看到的每一個位元、每一段碼、每一個百分比，都是當場用 JavaScript 跑演算法算出來的。

## 怎麼用

1. 在文字框輸入或貼上任何文字，或直接點「一段中文」「一段英文」「高重複文字」三個範例按鈕。
2. 往下看「字元頻率」——出現越多次的字元排越前面。
3. 「由下而上長出霍夫曼樹」會自動播放建樹動畫：發亮的節點代表「目前池子裡還在等的節點」，每一步合併兩個最小的、變成一個新節點放回池子。可以暫停／重播／直接跳到完整的樹。
4. 「碼表」列出每個字元的 0/1 編碼，注意碼的長度——常用字只要 2–3 個位元，罕見字要 10 幾個位元。
5. 「原始 vs 霍夫曼 vs 理論極限」把三種位元數並排，並算出壓縮率；下面還有霍夫曼平均碼長跟夏農熵的直接對比，差距通常小於 0.05 bit。
6. 最下面按「解碼還原驗證」，程式會把編碼後的位元串解回文字，逐字元比對是否跟原文完全相同。

## 為什麼有趣、為什麼實用

- **常用字自動變短，這不是人工設計，是演算法自己「長」出來的。** 你不用告訴它哪個字重要，它從頻率資料自己推出最佳的碼長分配。
- **壓縮率非常接近理論極限。** 夏農熵是任何「逐字元」編碼都不可能突破的下界；霍夫曼碼的平均長度離這個下界通常只差 0.02～0.05 bit（實測數字見下方「方法論」）。
- **中文、英文、高重複文字的壓縮率差很多**，這件事本身就很說明問題：文字的「可預測性」越高（重複越多、分布越不平均），能壓縮的空間就越大——這正是資訊理論的核心直覺：意外的東西才需要花位元去描述，理所當然的東西不用。
- **這是 ZIP、JPEG、MP3、PNG、傳真機、乃至 HTTP 壓縮背後其中一塊基礎積木**，理解它，等於理解你每天用的東西怎麼變小的。

## 運作原理／方法論

### 演算法本身

- **建頻率表**：把輸入文字拆成 Unicode 字元（用 `Array.from` 逐字元拆解，正確處理 emoji 等由代理對組成的字元），統計每個相異字元出現次數。
- **真的實作優先佇列**：手刻一個二元最小堆積（binary min-heap，見頁面原始碼 `HuffMinHeap`），支援 `push`／`pop`，權重相同時用「先出現的字元先處理」的規則穩定排序，確保每次執行結果可重現。
- **建樹**：重複「彈出兩個最小的節點、合併成一個新節點（權重相加）、推回堆積」，直到只剩一個根節點。共執行 n−1 次合併（n 為相異字元數），每次 push/pop 是 O(log n)，**整體時間複雜度 O(n log n)**。合併過程全程記錄下來，用來播放建樹動畫。
- **產生前綴碼**：從根節點往下走訪，往左走記一個 0、往右走記一個 1，走到葉節點就是該字元的完整編碼。只有一種字元的退化情形，直接指定碼為 `"0"`（否則會產生空字串碼，無法在位元流裡界定字元邊界）。
- **編碼／解碼**：編碼就是把每個字元換成它的碼、串接起來；解碼是從樹根出發，位元是 0 就往左、1 就往右，撞到葉節點就吐出一個字元、跳回根節點重新走——完全不需要分隔符號。
- **夏農熵**：H(X) = −Σ p(xᵢ)·log₂p(xᵢ)，用每個字元的實際出現機率算出。這是「逐字元編碼」理論上能達到的最短平均長度，霍夫曼碼保證平均長度落在 `[H, H+1)` 區間內（只有一種字元的退化情形是等號 `H+1`，因為此時 H=0 但仍需要至少 1 bit 才能界定字元邊界）。
- **固定長度基準**：用 `⌈log₂(相異字元數)⌉` 位元／字元當作「原始」基準——這是最公平的比較方式：同一個字母表，不用霍夫曼的話，最少需要幾個位元才能不重複地表示每個字元。

### 建樹動畫的版面配置

樹的畫面座標不是隨便排的：

- **X 軸**：對最終建好的樹做中序走訪（in-order traversal），依序把 X 座標分給每個葉節點，內部節點的 X 則取左右兩個子節點 X 的中點。這是課本上「畫二元樹不會有線交叉」的標準技巧。
- **Y 軸**：用節點在樹裡的實際深度（root 深度 0，越往下越深）決定，不是用合併發生的時間順序——這樣才會是一棵「正常形狀」的樹，而不是每次合併都往上疊一層的畸形樓梯。不過動畫仍然照著真實的合併順序（時間軸）逐步播放，所以「先出現的子節點一定先畫出來、父節點永遠後出現」這個由下而上的因果關係，跟畫面呈現是分開處理但結果一致的。
- 相異字元數很多或機率分布很極端（實測過 300 種相異符號、以及刻意用費波那契數列構造的病態最深樹）時，畫布會變得很寬或很高，容器可以左右／上下捲動，不會撐壞版面。

### 三個踩過的 SVG 地雷（已避開）

1. 樹的邊用漸層色（`stroke:url(#grad)`）——若不指定 `gradientUnits="userSpaceOnUse"` 並給實際座標，瀏覽器預設用 `objectBoundingBox`，會讓水平或垂直的直線（邊界框高或寬為 0）整條消失不見。這裡的樹很容易長出垂直邊，已經明確加上 `userSpaceOnUse` 加實際座標避開。
2. 邊的發光效果（`filter:drop-shadow`）掛在包住所有邊的 `<g>` 群組上，而不是每一條 `<path>` 各自掛——理由同上，零面積的路徑會讓逐路徑的濾鏡整條不繪製。
3. 每一條邊的「畫線」動畫用 `stroke-dasharray`／`stroke-dashoffset`：先把 `transition` 設為 `none` 並把線隱藏、強制觸發一次重排（讀取 `getBoundingClientRect()`)、下一個 `requestAnimationFrame` 才打開 transition 並把 `dashoffset` 設回 0——確保每次重播（暫停、重播、跳過動畫再重來）都能可靠地重新畫一次，不會卡在中途或直接跳過動畫效果。
4. 節點的縮放進場動畫用 CSS `transform`，但每個節點的位置是用 SVG 的 `transform="translate(...)"` 屬性設定的——如果直接在同一個元素上疊加 CSS `transform: scale(...)`，CSS 屬性會整個蓋掉、而不是疊加 SVG 的 `translate` 屬性，導致節點瞬間跳到錯誤座標。解法是套一層內層 `<g>`：外層只負責定位（SVG 屬性、永遠不被 CSS 動到），內層只負責縮放淡入（純 CSS transform，不帶任何定位資訊）。

## 為什麼前綴碼不用分隔符就能唯一解碼

霍夫曼碼保證「沒有任何一個字的碼，是另一個字的碼的開頭」。解碼時從樹根出發，位元是 0 往左、1 往右，一旦走到葉節點就代表剛好湊滿一個完整的字元碼——因為前綴碼的性質，不可能存在「走到一半、卡在某個內部節點、卻分不清該不該當作一個字元結束」的情況。所以整段位元流貼在一起，也能無歧義、逐字元地解回原文，完全不需要逗號、空格或任何分隔符號。

## 為什麼霍夫曼是最佳的「逐字元」編碼

可以用交換論證（exchange argument）說明：在任何最佳前綴碼裡，出現次數最少的兩個符號一定深度最深、而且互為兄弟節點——如果不是，把它們换到更淺的位置只會讓總長度變得更長，不會更短。霍夫曼演算法每一步都精確地做這件事：找出當下最小的兩個節點、把它們變成兄弟、合併成一個新符號繼續遞迴。因此每一步都不會偏離最佳解的路徑，最終蓋出來的樹就是全域最佳解。這是 David Huffman 在 1952 年論文裡證明的核心定理。

## 為什麼它仍輸給算術編碼與 LZ77／LZ78

霍夫曼碼有一個無法迴避的限制：**每個符號至少要花掉 1 個完整位元**，就算某個符號出現機率高達 90%、理論上只需要約 0.15 bit 也一樣，因為前綴碼的碼長必須是整數。

- **算術編碼（arithmetic coding）**把整段訊息編碼成一個介於 0 和 1 之間的單一小數，不受「整數位元」限制，平均長度能真正逼近夏農熵，在符號機率很不平均時明顯優於霍夫曼。
- **LZ77／LZ78 這類字典法**看的不是單一符號的出現頻率，而是「重複出現的字串」——多個符號組成的上下文規律（例如英文裡的 "the"、程式碼裡固定的縮排與關鍵字）。這是霍夫曼這種靜態、逐字元的機率模型天生看不到的高階重複，所以像 ZIP／DEFLATE 這類實務壓縮格式，通常是先用 LZ77 抓字串重複、再用霍夫曼把結果壓最後一段，兩種技術截長補短。

## 這是真的嗎（歷史故事查證）

頁面故事段落裡的每一個情節都查證過，主要來源如下：

- **1951 年，MIT 資訊理論課，二選一的作業**：David Huffman 是 MIT 電機系研究生，修 Robert M. Fano 開的資訊理論研究生課程。Fano 讓學生二選一：期末考，或是一篇學期報告，題目是找出最有效率的二進位符號編碼法。Huffman 選了報告。（MAA Convergence〈Discovery of Huffman Codes〉；Scientific American 1991 年 9 月號 Gary Stix 撰寫的 Huffman 人物側寫）
- **Fano 沒告訴學生這是一個公開難題**：Fano 當時正與 Claude Shannon 一起研究同一個問題，但沒有告訴班上這其實是他們自己都還沒解決的問題。Huffman 後來受訪表示，若當初知道是老師和夏農都卡關的題目，他大概不會去嘗試。（Scientific American 1991；MAA Convergence）
- **正要放棄、丟筆記的那一刻**：Huffman 在這題上耗了數月都無法證明任何方法是最佳解，一度決定放棄、改去讀書準備期末考，就在丟筆記的那一刻想通了由下而上的合併法。原話：「It was the most singular moment of my life. There was the absolute lightning of sudden realization.」（Scientific American 1991，Gary Stix 專訪原文）
- **由下而上勝過由上而下**：Fano 與 Shannon 當時的方法（後稱 Shannon–Fano coding）是由上而下建樹，不保證找到最佳解；Huffman 反過來由下而上、每次合併當下最小的兩個，被證明永遠是最佳解。Fano 看到答案時說「Is that all there is to it!」（Scientific American 1991；Quanta Magazine 2023 年關於資料壓縮的專文，文中並用 "schoolroom" 一詞實際示範 Fano 法比 Huffman 法多花 1 個位元）
- **1952 年發表，唯一的獎勵是免修期末考**：論文〈A Method for the Construction of Minimum-Redundancy Codes〉發表於 *Proceedings of the IRE*, 40(9): 1098–1101, September 1952。Huffman 當時是 MIT 的 Sc.D.（工程科學博士）研究生，25 歲。（Wikipedia〈David A. Huffman〉；Scientific American 1991）

## 測試方式與實測數字

依照專案慣例做過三層測試：

1. **語法檢查**：把 `index.html` 內所有 `<script>` 內容抽出成獨立 `.js`，跑 `node --check` 全部通過。
2. **演算法斷言**（把核心演算法逐字複製成獨立 Node 腳本後執行，避免抄寫誤差，共 3865 項斷言全數通過）：
   - **Round-trip 無損還原**：中文範例、英文範例、高重複文字、單一字元重複 20 次、極短字串（1 個字元／2 個字元）、空字串、中英數混合、含 emoji（代理對）字元——每一組編碼後解碼都逐字元完全等於原文。
   - **平均碼長界限**：對每一組非空文字驗證 `平均碼長 ≥ 夏農熵`，且相異字元數 ≥ 2 時嚴格 `< 熵 + 1`；只有一種字元的退化情形驗證等號邊界 `平均碼長 = 1 = 熵(0) + 1`。
   - **前綴碼性質**：對碼表裡每一對碼做逐對檢查，確認沒有任何一個碼是另一個碼的前綴。
   - **邊界情況不崩潰**：空字串、單一字元（重複很多次）都能正常跑完、不拋例外。
   - 額外驗證了優先佇列（min-heap）本身彈出順序永遠遞增、以及樹版面配置演算法在「300 種相異符號的長文字」與「刻意用費波那契數列構造的病態最深樹（20 個符號、深度 19）」這兩個極端情境下，所有節點座標皆為有限數值、父節點一律比子節點更靠近樹根。
3. **實測壓縮數據**（節錄自測試腳本輸出）：

   | 範例 | 相異符號數 | 平均碼長 (bit) | 夏農熵 (bit) | 差距 (bit) | 壓縮率 |
   |---|---|---|---|---|---|
   | 中文範例 | 50 | 5.4429 | 5.4182 | 0.0246 | 9.29% |
   | 英文範例 | 29 | 4.2442 | 4.2093 | 0.0349 | 15.12% |
   | 高重複文字 | 7 | 1.2105 | 0.6025 | 0.6080 | 59.65% |
   | 中英數混合 | 19 | 3.8750 | 3.8400 | 0.0350 | 22.50% |

   可以看到：不管文字種類，霍夫曼平均碼長離夏農熵理論下界永遠只差一點點（多數情況差距在 0.02～0.05 bit 之間）；而「壓縮率」則高度取決於文字本身的重複性——高重複文字能壓掉近六成，天然語言文字受限於相異符號較多、分布相對平均，壓縮率沒那麼誇張，但仍穩定省下 9～23% 的位元，且都精準逼近各自的理論極限。
4. **無外部資源／localStorage 前綴檢查**：`grep` 確認頁面裡唯一的 `http://` 字串是 SVG 命名空間 `http://www.w3.org/2000/svg`（規格允許），沒有任何 `fetch`／`XMLHttpRequest`／`<link>` 外部資源；`localStorage` 只用到一個 key，且已加專案前綴。
